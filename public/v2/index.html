Problem

Manual OTDR triage is a structural bottleneck.

Most fiber operations teams still correlate alarms by hand, interpret traces manually, and translate findings into dispatch notes under time pressure. The result is slow, inconsistent, and dependent on a small number of senior engineers.

  • Signal overload. High event volume from multiple telemetry sources obscures actionable incidents.
  • Knowledge bottleneck. OTDR trace interpretation depends on specialists who are not always on shift.
  • Slow handoff. Field teams receive delayed or incomplete fault localization data, extending MTTR.
  • Inconsistent output. Root cause narratives vary by engineer experience, shift, and workload.

Architecture

Five-stage deterministic and agentic pipeline.

Deterministic stages handle normalization, deduplication, and policy enforcement. Agentic stages interpret traces, compare baselines, and produce structured root cause proposals with confidence scoring and evidence citations.

Stage 01

Ingest

Capture SNMP traps, REST telemetry, and streaming data into a normalized event envelope.

  • multi-vendor
  • deduplication
  • burst guard
Stage 02

Normalize + Enrich

Attach topology metadata, circuit context, maintenance windows, and historical trace baselines.

  • digital twin
  • path mapping
  • dependency graph
Stage 03

Trace Analysis

Compare live OTDR traces against baseline signatures. Detect breaks, reflections, and attenuation drift.

  • anomaly detection
  • distance estimate
  • pattern fusion
Stage 04

Reason + Score

Generate root cause hypotheses with confidence scores and explicit evidence citations.

  • explainable RCA
  • confidence threshold
  • FP suppression
Stage 05

Deliver + Act

Ship structured incident packages to messaging platforms, ticketing systems, and dispatch workflows.

  • webhook output
  • PDF report
  • SLA escalation

First alert in ~30 seconds. Full incident report with root cause, confidence score, and recommended action in ~5 minutes. All model decisions are logged and auditable.

Design Principles

Vendor-agnostic. Model-agnostic. No lock-in.

Hikari integrates with your existing telemetry infrastructure without replacing it. The analysis layer is decoupled from both the data source and the inference backend.

Vendor-agnostic telemetry

Normalized connectors abstract vendor-specific protocols into a unified event format. Add new telemetry sources without modifying the analysis pipeline.

  • SNMP, REST, and streaming connector framework
  • Vendor-specific translation at the edge
  • No dependency on proprietary telemetry formats

Model-agnostic inference

The reasoning layer supports swappable inference backends. Deploy with your preferred model provider or run fully local with open-weight models.

  • Abstracted inference interface
  • No hard dependency on any single model provider
  • Local, private cloud, or hosted — your choice
